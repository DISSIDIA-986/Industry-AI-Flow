## 一、评审背景与目的

### 1.1 为什么需要第二轮评审

第一轮评审（v1，同日）对系统进行了初步检视，识别出 3 个 P0 问题：RAG 准确率不足、部署流程复杂、文件上传安全缺陷。但该轮评审主要基于 **功能层面的黑盒观测**，未深入代码实现。

### 1.2 评审目标

- 评估系统是否具备 **部署或交付测试** 的必要条件
- 识别 v1 评审遗漏的 **深层架构与安全缺陷**
- 基于 2026 年 AI 开源生态现状，给出 **务实可落地** 的改进建议
- 精选 **最多 5 个 Must-Change** 项，为管理层提供清晰的决策依据

### 1.3 与 v1 评审的关键差异

| 维度 | v1（第一轮） | v2（本轮） |

## 四、各专家视角：深度发现

### 4.1 AI/RAG 专家视角

#### 发现 1：⚠️ 会话记忆系统形同虚设（P0 级）

**现象**: 系统实现了一套精心设计的三层记忆架构——短期窗口记忆、LLM 摘要记忆、长期向量记忆（`backend/services/memory/` 下 5 个文件），但 **RAG 主管线从未调用它**。

**代码证据**: `backend/services/rag_engine.py` 中的 `SimpleRAG.query()` 方法是 RAG 的核心入口。阅读该方法全部代码，流程为：

```
查询 → 自适应权重 → 混合检索 → 重排序 → 构建 Prompt → LLM 生成 → 安全检查 → 返回
```

其中构建 Prompt 阶段只注入了检索到的文档片段，**没有任何对 `ConversationMemoryManager.build_memory_payload()` 的调用**。用户的多轮对话上下文被完全丢弃。

**影响**: 用户每次提问都是独立的，系统无法理解"上一个问题提到的那个规范"等指代。这对于建筑行业知识问答是核心功能缺失。

**修复方案**: 在 `SimpleRAG.query()` 中集成 `build_memory_payload()`，将短期记忆和相关长期记忆注入 system prompt。预估 2-3 天。

#### 发现 2：⚠️ BM25 索引不随文档更新（P0 级）

**现象**: `backend/services/retrieval/hybrid_search.py` 中，BM25 索引在首次搜索时一次性从数据库加载全部 chunks 到内存构建索引。**之后新增的文档永远不会进入 BM25 索引**。

**影响**: 用户上传新文档后，混合搜索中的关键词检索部分搜不到新内容，只有向量搜索能覆盖。这意味着 RRF 融合的效果被严重削弱。随着文档增多，BM25 与向量搜索的覆盖范围严重不对齐。

此外，全量加载到内存的方式在大规模文档集下不可扩展（1 万+ 文档时内存占用会显著增长）。

**修复方案**: 
- 短期：增加 BM25 索引重建触发机制（文档上传后自动/定时重建）
- 中期：考虑改用 PostgreSQL 原生全文搜索（`ts_vector` + `ts_query`），已有 GIN 索引基础
- 预估 3-5 天

#### 发现 3：无 RAG 评估框架

**现象**: `tests/evaluation/ragas_evaluation.py` 是一个骨架文件，无实际实现。系统没有任何检索质量指标（MRR、NDCG、Recall@K）或端到端评估。

**特别说明**: v1 评审声称"RAG 准确率为 75%"，但 **这个数字缺乏任何评估框架支撑**。在没有评估能力的情况下，任何准确率数字都是猜测，不应作为决策依据。

**影响**: 无法度量质量 = 无法验证改进 = 无法交付测试。这是交付测试的前置条件。

**修复方案**: 
- 引入 RAGAS（已是 2026 年 RAG 评估的事实标准，GitHub 20k+ star）或 DeepEval
- 构建包含 50-100 个建筑领域 Q&A 对的评估数据集
- 预估 1 周（含数据集标注）

#### 发现 4：Groundedness 检查仅为词汇匹配

**现象**: 系统中存在两个 groundedness checker 实现：
- `backend/services/safety/groundedness_checker.py` — token 重叠率计算
- `backend/safety/groundedness_checker.py` — 基于正则的声明提取 + 字符串匹配

两者均为 **纯词汇级别** 的检查，没有语义理解能力。代码注释中提到了 NLI（自然语言推理）模型，但实际未使用。

**影响**: 建筑行业涉及安全关键信息（结构荷载、材料规范、施工安全等），仅靠词汇匹配判断答案是否有依据远远不够。LLM 可以通过改换措辞轻松"通过"词汇检查但输出错误信息。

**2026 生态建议**: 引入轻量 NLI 模型（如 `cross-encoder/nli-deberta-v3-small`，约 180MB），既能提高检测精度，又不会大幅增加延迟。

#### 发现 5：其他 RAG 层面问题

| 问题 | 详情 |
|------|------|
| **无元数据过滤** | 向量搜索和 BM25 均无法按文档类型、日期等过滤，所有 chunks 无差别搜索 |
| **Chunking 参数不一致** | `chunk_text()` 硬编码 512/128，`settings` 默认 300/50，两条路径产出不同大小的 chunk |
| **IVFFlat 索引未调参** | 未指定 `lists` 参数，小数据集下性能次优；未提供 HNSW 选项 |
| **无查询扩展** | 无 HyDE、多查询分解、同义词扩展等技术 |
| **Embedding 模型偏旧** | nomic-embed-text-v1.5 发布于 2024 年中；2026 年已有 BGE-M3、Jina v3 等更优选项 |

### 4.2 项目内部架构师视角

#### 发现 1：⚠️ 应用启动存在崩溃风险（P0 级）

**问题 A — 模块级 Agent 构造**:

`backend/agents/unified_agent.py` 在文件末尾（约 L481-482）直接执行：
```python
unified_agent = create_unified_agent()
unified_orchestrator = UnifiedOrchestrator(unified_agent)
```

这段代码在 **模块被 import 时立即执行**。如果 LLM 服务（Ollama/智谱）不可用、或任何依赖包缺失，整个模块导入失败，导致 FastAPI 应用启动崩溃。

**问题 B — 全局变量无线程安全**:

`backend/main.py` L61-63 定义了三个可变全局变量：
```python
rag_engine = None
unified_orchestrator = None
code_executor = None
```

通过懒加载模式在首次请求时初始化，但 **没有使用锁（Lock）保护**。在并发请求场景下，多个线程可能同时检测到 `None` 并各自创建实例，造成资源浪费和潜在的状态不一致。

**影响**: LLM 服务不可用时系统直接无法启动（而非优雅降级）；并发首次请求可能出现竞态条件。

**修复方案**: 
- 将模块级构造改为 FastAPI `lifespan` 中的延迟初始化，异常时记录日志并置为 `None`
- 全局变量增加 `threading.Lock` 保护
- 预估 2-3 天

#### 发现 2：main.py 单体问题（P1 级）

`backend/main.py` 有 **857 行**，包含：
- Pydantic 请求/响应模型定义
- 文件处理工具函数
- 大量内联端点处理函数（与 `backend/api/` 下的路由模块功能重叠）
- 全局变量和初始化逻辑

这不仅是代码风格问题，更导致：
- 路由逻辑分散在两处，维护者难以确定"这个端点定义在哪里"
- 修改一处可能遗忘另一处的对应端点
- 单元测试难以隔离

**修复方案**: 将 `main.py` 中的端点全部迁移到 `api/` 下对应模块，`main.py` 仅保留 app 创建、中间件注册、路由挂载。目标控制在 100 行以内。预估 2-3 天。

#### 发现 3：Config 反模式（P1 级）

`backend/config.py` 在每个字段默认值中手动调用 `os.getenv()`：
```python
postgres_host: str = os.getenv("POSTGRES_HOST", "localhost")
```

这 **绕过了** pydantic-settings `BaseSettings` 的核心能力（自动读取环境变量、类型转换、`.env` 文件支持、字段验证）。相当于引入了 pydantic-settings 但只当普通 dataclass 使用。

**修复方案**: 移除所有 `os.getenv()` 调用，改为：
```python
postgres_host: str = "localhost"  # pydantic-settings 自动读取 POSTGRES_HOST
```
配合 `model_config = SettingsConfigDict(env_file=".env")` 启用 `.env` 文件支持。预估 1 天。

#### 发现 4：其他架构问题

| 问题 | 详情 |
|------|------|
| **`_get_llm()` 重复 3 次** | 相同的 LLM 实例化函数出现在 `rag_agent.py`、`code_analysis_agent.py`、`unified_agent.py` |
| **废弃的生命周期钩子** | 使用已废弃的 `@app.on_event("startup")`，应迁移到 FastAPI `lifespan` context manager |
| **无数据库迁移工具** | 依赖 `CREATE TABLE IF NOT EXISTS`，无法处理列类型变更或删除；Alembic 在依赖中但未使用 |
| **混合同步/异步 DB 驱动** | psycopg3 (sync) 和 asyncpg (async) 并存，增加连接池管理复杂度 |
| **f-string 日志** | 多处使用 `logger.info(f"...")` 而非 `logger.info("%s", ...)`, 即使日志级别禁用也会构造字符串 |

### 4.3 安全专家视角

#### 发现 1：⚠️ 代码执行沙箱 AST 黑名单可绕过（P0 级）

`backend/services/code_executor.py` 使用 Python AST 分析来检测危险代码（`exec`、`eval`、`os.system`、`subprocess` 等）。但这种 **静态黑名单** 存在多种已知绕过方式：

```python
# 绕过方式 1：getattr
getattr(__builtins__, 'eval')('malicious_code')

# 绕过方式 2：importlib
import importlib
os = importlib.import_module('os')

# 绕过方式 3：字符串拼接
func = globals()['__builtins__']['ev' + 'al']
```

**影响**: 代码执行是用户可直接触发的功能。虽然有 Docker 容器隔离作为第二层防护，但容器配置也存在问题（见下一条），形成 **双层防护均不完善** 的局面。

**修复方案**: 
- AST 检查改为 **白名单模式**（只允许已知安全的模块和函数），而非黑名单
- Docker 容器增加 `read_only=True`（只读根文件系统）
- 考虑增加 seccomp profile 限制系统调用
- 预估 3-5 天

#### 发现 2：⚠️ 认证与授权默认关闭（P0 级）

`backend/config.py` 中：
```python
require_api_key: bool = os.getenv("REQUIRE_API_KEY", "false").lower() == "true"
# REQUIRE_USER_AUTH 也默认 false
```

这意味着 **默认部署状态下，所有 API 端点完全开放**，包括代码执行、文档上传、LLM 调用等敏感操作。

**影响**: 任何知道服务地址的人都可以执行任意 Python 代码、上传文件、消耗 LLM 配额。即使是内部测试部署也不应如此。

**修复方案**: 
- 将 `require_api_key` 默认值改为 `true`
- 首次启动时自动生成并输出一个随机 API key
- 仅开发环境通过显式配置 `REQUIRE_API_KEY=false` 关闭
- 预估 1 天

#### 发现 3：其他安全问题

| 问题 | 详情 | 严重度 |
|------|------|--------|
| **文件上传无内容校验** | 仅校验文件扩展名，无 magic number 检测、无 MIME 类型验证、无病毒扫描 | P1 |
| **容器无只读根文件系统** | Docker 沙箱未设置 `read_only=True`，攻击者可在容器内写文件 | P1 |
| **无 seccomp/AppArmor** | 容器未限制系统调用，增加逃逸风险 | P2 |
| **JWT 密钥管理** | 密钥来自环境变量，无轮换机制，无 KMS 集成 | P2 |

### 4.4 DevOps 专家视角

#### 发现 1：⚠️ 无完整部署管线（P1 级）

**Docker Compose 不完整**: `docker-compose-postgres.yml` 仅包含 PostgreSQL 容器。缺少：
- 后端应用容器
- Ollama / LLM 服务容器
- 前端容器
- Redis 缓存（配置中提及但未部署）

**无 CI/CD**: 项目中没有 GitHub Actions、GitLab CI 或任何 CI/CD 配置文件。代码质量门禁（`make test-phase1-gate` 等）只能手动运行。

**手动部署步骤过多**: 根据 `docs/development/status/active/DEPLOYMENT_STATUS.md` 记录，完成一次部署需要 10+ 手动步骤（安装 pgvector、创建扩展、部署 venv、配置环境变量、初始化数据库、启动服务…）。

**影响**: 
- 无法持续集成 = 合并代码时没有自动质量保障
- 手动部署 = 每次部署都可能因人为失误失败
- 不满足"交付测试"的最低基础设施要求

**修复方案**: 
- 编写完整的 `docker-compose.yml`（包含 app + postgres + ollama）
- 创建基础 GitHub Actions 工作流（lint → test → build）
- 编写 `scripts/deploy.sh` 一键部署脚本
- 预估 1 周

#### 发现 2：其他 DevOps 问题

| 问题 | 详情 |
|------|------|
| **沙箱镜像无构建管线** | `luncheon/code-analysis:v1.0` 镜像名硬编码在 config 中，但 repo 中无该镜像的 Dockerfile 构建流程 |
| **监控基础设施缺失** | 代码中有 Prometheus 埋点（`prometheus-fastapi-instrumentator`），但无 Prometheus/Grafana 部署配置 |
| **无健康检查编排** | 各组件有 health check 端点，但无统一的启动顺序和依赖等待机制 |
| **日志仅写文件** | 审计日志写本地文件，无集中式日志收集（ELK/Loki） |
| **备份无自动化** | 数据库备份依赖手动 `pg_dump` |

### ⬛ Must-Change #1：修复 RAG 管线核心缺陷

| 维度 | 内容 |
|------|------|
| **级别** | P0 — 核心产品功能有 bug |
| **问题** | ① Memory 系统未接入 RAG 管线（对话记忆形同虚设）<br>② BM25 索引不随文档更新（新文档搜不到） |

## 六、2026 开源生态参考（务实取舍）

作为评审的一部分，我们调研了截至 2026 年 2 月的相关开源项目，评估了"可借鉴"与"不适合"的边界：

### 可借鉴（低成本引入）

| 项目/组件 | 推荐用法 | 引入成本 |
|-----------|---------|----------|
| **RAGAS** (GitHub 20k+ ⭐) | 直接集成为 RAG 评估框架，替代空壳 `ragas_evaluation.py` | 低（pip install + 编写评估脚本） |
| **BGE-M3** (BAAI) | 评估替换 nomic-embed-text-v1.5 的可行性，支持多语言+多粒度 | 中（需要 GPU、需做对比实验） |
| **PostgreSQL 全文搜索** | 替代 BM25 in-memory 索引，利用已有 GIN 索引 | 低（SQL 层改动） |
| **NLI 轻量模型** | `cross-encoder/nli-deberta-v3-small` 替代词汇匹配 groundedness | 低（~180MB 模型） |

### 值得关注但当前不建议引入

| 项目 | 理由 |
|------|------|
| **RAGFlow** (InfiniFlow, 15k+ ⭐) | 文档解析能力强大，但整体替换现有管线成本过高；可参考其 chunking 策略 |
| **Dify** | Workflow 可视化编排优秀，但本项目已有 LangGraph 管线，替换价值不大 |
| **ColBERT v2** | 检索质量提升明显，但部署复杂度高（需要独立索引服务），Capstone 阶段性价比不高 |
| **GraphRAG** (Microsoft) | 适合复杂多跳推理场景，但本项目主要是单跳知识问答，引入后复杂度剧增 |
| **vLLM / TGI** | LLM 推理优化框架，适合高并发生产环境；当前 Ollama 已满足开发/演示需求 |

### 总体原则

> **修复优先于替换，借鉴优先于引入，可落地优先于前沿性。**
> 
> 当前系统已有完整的 RAG 管线、Agent 框架、安全机制。问题不在于架构设计方向，而在于关键环节的实现缺陷。优先修复已有代码中的 bug 和安全漏洞，而非用外部系统替换。
