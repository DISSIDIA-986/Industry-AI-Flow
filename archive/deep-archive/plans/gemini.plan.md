# Gemini's Plan for an Agentic AI Workflow Platform

This plan outlines the architecture, technology choices, and implementation steps for building a unified AI workflow platform, as per the requirements in `tmp/prompt.ai-workflow.md`.

## 1. Architecture Design

The platform will be built on a modular, microservices-based architecture to ensure scalability, maintainability, and flexibility. This design separates the core functionalities into distinct layers, communicating via a central API Gateway and an asynchronous message queue.

- **Presentation Layer:** A React-based web interface for user interaction, administration, and data visualization.
- **API Gateway:** A single, unified entry point for all frontend requests. It will handle authentication, routing, and rate limiting.
- **Service Layer (Core Microservices):**
    - **Data Ingestion Service:** Handles connections to various data sources (local files, URLs, databases). It will integrate the `deepseek-ocr` service for document parsing.
    - **Processing Pipeline Service:** Manages the document lifecycle: cleaning, chunking, and generating embeddings. It places processed data onto the message queue.
    - **RAG Service:** The core retrieval and generation engine. It queries the vector database, applies re-ranking, and interacts with the LLM to generate answers. Built using LangGraph to manage the complex flow.
    - **Agentic Service:** Contains specialized AI agents. The first agent will be for advanced data analysis, with the ability to generate and execute Python code in a secure environment.
    - **Code Execution Service:** A secure, isolated environment (using Docker) for running AI-generated code.
- **Data & Persistence Layer:**
    - **Relational DB (PostgreSQL):** Stores user data, project configurations, and application metadata.
    - **Vector DB (Qdrant):** Stores and indexes document embeddings for fast retrieval.
    - **Conversational Memory (Redis):** Manages session history and conversation context for a fluid user experience.
- **Asynchronous Task Queue (RabbitMQ/Kafka):** Decouples long-running tasks like document ingestion and embedding from the main application flow, improving responsiveness.

## 2. Technology Selection

This selection prioritizes modern, open-source, and enterprise-ready solutions that align with your experience and the project goals.

- **Workflow Orchestration:** **LangGraph**. Its graph-based structure is perfect for building complex, stateful, and cyclical agentic workflows, offering more control than linear chains.
- **LLM Model:** **Llama-3.1**. It offers a great balance of performance, a large context window, and strong community support. For the specialized data analysis agent, we can route to **Qwen2.5-coder** if it shows superior performance on coding tasks.
- **Embedding Model:** **nomic-embed-text-v1.5**. It is a top-performing open-source model, crucial for high-quality retrieval.
- **ReRanker:** **bge-reranker-base-v2**. A strong, efficient model to improve the relevance of retrieved documents before they are passed to the LLM.
- **Vector Database:** **Qdrant**. Chosen for its performance, advanced filtering capabilities, and scalability. For users preferring a fully self-hosted solution within their existing infrastructure, **PostgreSQL with pgvector** will be supported as a strong alternative.
- **OCR Engine:** **deepseek-ocr**. Leveraging your positive experience and its proven effectiveness.
- **Relational Database:** **PostgreSQL**. Its stability, feature set, and the `pgvector` extension make it the ideal choice.
- **Code Execution Environment:** **Docker**. Provides the necessary security and isolation for executing untrusted, AI-generated code.
- **Conversational Memory:** **Redis**. It is fast, scalable, and well-suited for caching and managing session data.
- **Frontend & Visualization:** **React** for the main application UI, with **Plotly** integrated for rendering dynamic data visualizations generated by the data analysis agent.

## 3. Main Implementation Steps

The project will be developed in iterative phases.

### Phase 1: Core RAG Pipeline
1.  **Infrastructure Setup:** Deploy PostgreSQL and Qdrant instances using Docker Compose.
2.  **Data Ingestion & Processing:**
    - Develop the Ingestion Service, integrating `deepseek-ocr`.
    - Build the processing pipeline for chunking documents and generating embeddings with `nomic-embed-text-v1.5`.
3.  **Build Core RAG Service:** Use LangGraph to create the RAG pipeline, orchestrating retrieval from Qdrant, re-ranking with `bge-reranker-base-v2`, and generation with Llama-3.1.
4.  **Initial UI:** Create a simple Streamlit or Gradio interface for rapid prototyping and testing of the RAG pipeline.

### Phase 2: Agentic Capabilities & Advanced Features
1.  **Secure Code Execution:** Develop the isolated Docker-based environment for running Python code.
2.  **Data Analysis Agent:** Build an agent that can understand data analysis requests, write Python code (using Pandas, Plotly), execute it via the Code Execution Service, and return results (tables or visualizations).
3.  **Integrate Conversational Memory:** Connect the RAG and Agentic services to Redis to maintain conversation history and context.

### Phase 3: Enterprise Readiness & Full-Fledged UI
1.  **API Gateway & Security:** Set up an API Gateway (e.g., using Nginx or Traefik) to manage access to the services. Implement user authentication and authorization.
2.  **React Frontend:** Develop the primary user interface in React, providing a polished experience for knowledge management, chat, and data analysis.
3.  **Deployment Packaging:** Containerize all services and create a comprehensive `docker-compose.yml` or Kubernetes manifests for easy on-premise deployment.

### Phase 4: Customization and Optimization
1.  **Fine-Tuning Pipeline:** Establish a workflow to collect user feedback and data.
2.  **LoRA Fine-Tuning:** Use the collected data to perform LoRA fine-tuning on the Llama-3.1 model to enhance its performance on specific vertical domains or tasks.
3.  **Continuous Evaluation:** Implement an evaluation framework to regularly test the performance of all components (retrieval, generation, agents) and guide further improvements.
